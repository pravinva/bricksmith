SEMANTIC MODEL REFRESH · FEB 2026 · v2.0

SEMANTIC MODEL REFRESH

Production-ready architecture for SLA-driven freshness across Fulton Hogan semantic models using metadata-driven controls, model-isolated Databricks Jobs, native table update triggers, and integrated Power BI task execution.

01 · PROBLEM

Current State & Impact

Inconsistent refresh behavior, workflow coupling, and limited observability increase risk for trust, cost, and operational outcomes.

Inconsistent Freshness

Same Gold data appears in multiple semantic models without standardized refresh policy or confidence in report currency.

Cadence Mismatch

Low-change data refreshes too often while near-real-time tables may lag, impacting operational decisions.

No Isolation

Without model-oriented workflow boundaries, one model failure can affect refresh reliability for unrelated models.

Outcome: elevated operational noise, avoidable cost, and reduced confidence in dashboard freshness.
02 · SOLUTION DESIGN

Validated Design Principles

Model-level refresh One workflow per model per cadence Native table update triggers SLAs as data Power BI task first Cost + health telemetry
Guiding principle: Databricks owns freshness orchestration and observability; semantic models own shaping and serving.
03 · METADATA CONTROL PLANE

Unity Catalog Metadata Schema

Configuration, SLA policy, and execution telemetry are all captured as Delta tables in edp_metadata.semantic_refresh.

Core Tables

  • semantic_models (model registry, owner, query mode)
  • table_refresh_config (cadence, trigger mode, SLA threshold)
  • refresh_execution_log (status, duration, trigger source)
  • refresh_cost_tracking (DBU + estimated USD)

Representative DDL Shape

CREATE CATALOG IF NOT EXISTS edp_metadata; CREATE SCHEMA IF NOT EXISTS edp_metadata.semantic_refresh; CREATE TABLE semantic_models (...); CREATE TABLE table_refresh_config (...); CREATE TABLE refresh_execution_log (...); CREATE TABLE refresh_cost_tracking (...);
04 · ORCHESTRATION

Model-Based Workflow Architecture

Workflow Pattern

  • AWM-DAILY, AWM-MEDIUM_FREQUENCY, AWM-TABLE_TRIGGERED
  • Finance-DAILY, Finance-MEDIUM_FREQUENCY, Finance-TABLE_TRIGGERED
  • Failure isolation by model and cadence

Task Sequence

  1. log_start
  2. refresh_model (Power BI task)
  3. log_complete
  4. capture_costs
  5. Owner/platform notifications on failure
05 · HEALTH & COST

Monitoring Scorecard

Reliability

Success rate by model over rolling 7 days.

SLA

Breach count vs threshold policy from metadata.

Cost

Cost per refresh and projected annualized spend.

Delivery surface: Databricks SQL dashboard over metadata and execution logs for operations and leadership visibility.
06 · DEPLOYMENT

Rollout Plan

Phase 1 · Pilot

Stand up metadata schema, seed pilot models, generate workflows, validate end-to-end refresh and telemetry.

Phase 2 · Scale

Onboard all models/cadences, tune SLA policy, operationalize dashboard and alerting.

Phase 3 · NRT

Enable table update triggers for validated domains and tune debounce/thresholds from metadata.

Final picture: metadata-driven policy + model-isolated jobs + native trigger/task execution + integrated observability.
